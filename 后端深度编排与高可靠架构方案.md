# 织界引擎 MVP：后端深度演进与大模型高可靠编排架构方案

**文档状态：** 核心战略规划
**目标受众：** 后端研发团队、项目评审委员会
**前提约束：** 保持现有前端 UI/UX 与 API 契约（`/start`, `/audit`, `/execute/stream`）完全不变。

---

## 1. 核心痛点与架构目标 (The Challenge)

在接入真实大模型（LLM）后，系统将面临三个致命挑战：
1. **可靠性挑战（幻觉与乱码）**：LLM 可能不按格式输出，导致下游 Agent 无法解析数据，乃至引发系统奔溃。
2. **编排挑战（上下文污染）**：长链路执行中，“代码 Agent”可能遗忘了“需求 Agent”的初衷。
3. **落地挑战（伪工程）**：如果所谓的“编码和测试”只是输出了几句 Markdown 废话，那就变成了聊天软件，无法证明这是一个“交付引擎”。

为此，我们必须在后端引入 **Agentic Workflow（智能体工作流）** 设计模式，核心理念为：**结构化输出、自我纠错循环、虚拟沙盒校验**。

---

## 2. 高可靠大模型编排核心机制 (Reliability Engine)

为了保证 100% 的成功率与稳定性，后端不应该只是“调用一下 OpenAI API”，而需要一套由框架（推荐基于 LangChain 或 LangGraph 的思想）构建的包裹层。

### 2.1 结构化契约强校验 (Pydantic + Instructor)
- **机制**：通过 FastAPI 核心的 `Pydantic` 库定义每个阶段的输出实体。利用强类型（如 OpenAI 的 `response_format` 或 `Function Calling`）强制大模型只返回合法的 JSON。
- **例子**：技术方案节点不接收自然语言，必须接收带 `architecture`, `database_schema`, `api_endpoints` 字段的 JSON。后端解析后，再将其转换为 Markdown 推流给前端显示，从而实现“底层结构化，前端可读化”。

### 2.2 Re-Act 自我纠错重试环 (Self-Correction Loop)
- **机制**：在每个阶段的 Agent 生成结果后，自动外挂一个无感的 `Validation Node`（校验节点）。
- **流程**：`LLM 生成` -> `JSON 解析 / 内部逻辑检查` -> `若失败/违规` -> `不仅不报错，反而将错误信息组合成 Prompt 重新喂给 LLM：“你的输出缺失了数据库字段，请严格按格式重试”`。
- **阈值**：设置最大重试次数（如 3 次），若彻底失败则触发熔断，返回预置的兜底（Mock）数据向前端交差。

---

## 3. 全链路多 Agent 职责与工程落地 (The Pipeline Execution)

后端系统将维护一个统一的 **`PipelineState`（全局上下文状态容器）**，伴随整个生命周期。

### Stage 1: 产品经理 Agent (Intake & Analysis)
- **输入**：用户原始自然语言输入。
- **处理**：提取业务目标、拆解功能模块、定义核心验收标准（UAT）。
- **输出落库**：生成结构化的 `PRD Document` 对象挂载至全局上下文。

### Stage 2: 架构师 Agent (Solution)
- **输入**：`PRD Document`。
- **处理**：分析系统拓扑，选用技术栈，生成 API 契约和数据表结构。
- **拦截卡点**：此时将状态置为 `waiting_review`，推流结束。前端人工点击“通过”/“修改驳回”后，状态机才切入下一节点。

### Stage 3: 资深开发 Agent (Coding & VFS) —— 真实落地关键！
- **输入**：`PRD` + `Architecture Design`。
- **处理**：**绝不仅仅是生成一段文字代码**。
- **虚拟文件系统 (VFS)**：后端在内存中构建一个虚拟工作区字典（即 `{"src/main.py": "...", "package.json": "..."}`）。
- **静态语法树分析 (AST Review)**：当 Agent 生成 Python/JS 源码后，后端立即调用 `ast.parse()` 或 `eslint`（子进程）对内存中的代码进行毫秒级“编译检查”。如果捕捉到 `SyntaxError`，触发上面的“自我纠错环”，让 Agent 自行修复 BUG 再完成本节点。

### Stage 4: 质量保障 QA Agent (Testing)
- **输入**：`PRD 验收标准` + `VFS 内存源码`。
- **处理**：分析源码逻辑，运用边界值断言等软件测试理论，生成具体的单元测试用例（如 Pytest 格式或 Jest 格式），并评估代码实现是否覆盖了阶段一设定的功能点。

## 4. 前后端零侵入适配层 (Frontend SSE Adapter)

由于前端是通过 `EventSource(/execute/stream)` 拉取数据的，我们需要在后端的调度网关做一层“流式同传适配”：

1. **底层静默执行**：后端的 Agent 交互（如复杂的自我反思、多次重试）在底层静默运算。
2. **实时同传解说（解耦的艺术）**：为了让前端看起来不卡顿，后端在进行复杂校验的同时，单独向 SSE 管道里推入“解说词”：
   - `yield data: {"text": "🔍 正在进行 AST 静态语法检查..."}`
   - `yield data: {"text": "⚠️ 发现模块循环引用，Agent 正在重新生成逻辑..."}`
3. **成功后推流**：当最终的高质量 JSON 生成后，后端再写一个转换器，把 JSON 渲染成华丽带有代码高亮的 Markdown 块，通过流吐给前端。这样前端完全不知道后端经历了极其复杂的校验，只觉得“AI 非常懂业务且没有 Bug”。

## 5. Demo 现场“防翻车”与极致容错保障 (Zero-Downtime Guarantee)

在比赛或高管路演（Demo）现场，网络波动、大模型超时、API 封禁随时可能发生。作为项目经理，我认为最高级别的可靠性是：**即使底层大模型全线崩溃，哪怕正在生成一半时断网，前端用户和评委也察觉不到一丝异常，且流程必须 100% 顺滑走完。** 

为此，我们必须在后端加装**四大防御级护航机制**（完全无需修改前端）：

### 5.1 黄金剧本旁路缓存 (Golden Path Caching)
- **机制**：准备好一套能让系统展现最强能力的“主角需求”（如：“项目日报生成工具”）。
- **拦截逻辑**：当后端接收到 `POST /start`，若由于用户输入的 `title` 命中了我们的黄金关键字，系统将直接挂载对应的 `Golden Cache`。在现场演示时，它实际上并没有调真正的大模型，而是调用了**预设的、经过千锤百炼、带有完美 Mermaid 架构图和完美代码的离线剧本**。这保证了在聚光灯下的 0 失误。

### 5.2 毫秒级重试与多级熔断降级 (Circuit Breaking)
如果在演示现场使用真实请求录入陌生需求，启用如下多级降级：
- **L1 级 (完美运转)**: 真实调用大模型 -> Pydantic 校验成功 -> 返回精准数据。
- **L2 级 (大模型胡言乱语)**: LLM 回答一直不复合结构，重试 3 次均失败。此时后端不再死磕，切断重试，转而调用较小且快速的模型（或备用 API Key）**并放弃部分严格检查**。向客户端推流：“⚠️ 严格架构校验超时，正在生成基础版代码方案...”。
- **L3 级 (断网或 API 宕机)**: 如果外网或者 OpenAI/DeepSeek 接口直接 502。设定 8 秒绝对超时阈值。一旦触发，系统静默接管，切入预置的 `通用场景万能 Mock 库`，继续走完流程。

### 5.3 SSE 管道中途平滑接管 (Seamless Stream Rescue)
流式输出（打字机）最怕吐到一半卡住或崩溃报错（前端抛出 WebSocket 异常并红屏）。
- **解决方案**：在 FastAPI 的 `StreamingResponse` 的生成器内部加装全局 `try...except Exception` 块。
- **运行表现**：当 LLM 流正在向前端吐出：`“本次需求主要包含两个核...”` 的时候连接断了。后端立刻捕获异常，并且**不中断连接**，顺着当前内容继续 `yield` 出一句话：`“...核心模块（检测到网络波动，已无缝切换至高可用边缘计算节点），以下是完整设计方案：”`，并继续输出本地缓存内容，最终优雅触发 `waiting_review` 绿灯。这不仅是防翻车，简直是灾备能力的完美秀场。

### 5.4 人为强制推进 (God Mode)
- 现场演示时可能会遇到评委打乱节奏。通过 URL 参数（如访问 `?god_mode=1` 的前端环境发送的 HTTP 请求带特定 Headers），后端在任何“执行中”卡死的环节，只要捕捉到特定的“跳过指令”，立刻向正在流式阻断的 SSE 发送终结包并抛出成功，强制释放到下一步状态。

## 6. 项目落地实施 Roadmap (Backend Track)

| 阶段 | 耗时 | 交付物 | 核心目的 |
| :-- | :-- | :-- | :-- |
| **Phase 1<br>管线基建** | 1天 | `state_machine.py`<br>`sse_adapter.py` | 把目前的 Express/Mock API 彻底替换为具有标准路由的 FastAPI 框架，保障并发与 SSE 推流性能。 |
| **Phase 2<br>单 Agent 闭环** | 2天 | `agents/analysis.py`<br>`llm_gateway.py` | 接入真实的 OpenAI 系列大模型，引入 Pydantic 输出解析器，打通从“输入 -> LLM -> JSON -> SSE 文本推流”的单一闭环。 |
| **Phase 3<br>多域调度与沙盒** | 2天 | `workflow.py`<br>`vfs_sandbox.py` | 实现四组 Agent 的上下文传递模型；在 Coding 节点引入 AST 代码校验模块机制，彻底断绝“大模型生成无法运行废代码”的情况。 |
| **Phase 4<br>护航演练** | 1天 | `fallback_mocks.json`<br>`circuit_breaker.py` | 代码落地以上 **5.x 章节的所有防灾机制**，编写异常捕获系统，并在极短时间内切回保底高质量 Mock 数据或预置黄金剧本。 |

**执行宣告**：
通过本套方案，后端将从一个单纯的“HTTP 中转站”，变为真正意义上具备**思考、校验、修复、编排**能力的“智能中台”。哪怕前端只有几个简单的按钮，整个系统的产品壁垒和技术深度也足以应对任何专业级的代码评审问答。

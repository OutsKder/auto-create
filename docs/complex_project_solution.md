# 复杂项目代码生成与测试进阶方案

随着项目复杂度的提升，仅仅让单个 Agent 盲目地一次性尝试编写代码并测试将面临巨大挑战（如上下文溢出、逻辑遗漏、依赖冲突等）。为了让我们的 `Coder Agent` 和 `Test Agent` 能够胜任真实且复杂的工程项目，我们需要进行系统性的改进。

## 一、 当前痛点分析
1. **缺乏规划**：Agent 接到需求后直接开始写代码，缺乏自顶向下的架构拆解和实现规划。
2. **依赖隔离与管理**：没有处理第三方依赖（如 `pip install` 或 `npm install`），复杂项目跑不起来。
3. **上下文迷失**：几十个文件的项目，Agent 很容易忘记其他文件的接口长什么样。
4. **单步调试效率低**：每次完整的 LLM 交互只为了修一个小拼写错误，浪费 Token 且速度慢。

## 二、 进阶改进方案 (The Blueprint)

### 1. 引入“任务拆解与计划”（Plan & Execute 机制）
**对于 Coder Agent：**
- 在开始使用 `write_file_tool` 之前，强制要求 Agent 先输出一份“开发计划（Plan）”。
- 复杂需求往往包含：初始化目录结构、配置依赖文件（`requirements.txt`）、编写核心逻辑、编写辅助模块。
- **落实方案**：在 System Prompt 中强调：碰到复杂工程，必须先建立目录、分析步骤，并使用注释的方式规划进度。

### 2. 强化“环境与寻根”能力 (Environment & Context)
**对于 Coder & Test Agent：**
- **文件检查与感知**：充分利用已经加入的 `read_file_tool` 和 `list_dir_tool`，强调遇到不熟悉的模块时，先看它的源码，不要去猜接口。
- **依赖安装工具**：通过 `run_command_tool` 赋予沙箱安装依赖的能力（比如允许执行 `pip install xxx`）。

### 3. 多模态/长文本自愈（Auto-Healing with Context）
**对于 Test Agent：**
- 测试失败时，不能仅仅把错误丢给大脑，应该附带：最近修改了哪些文件？错误堆栈长什么样？
- **落实方案**：在错误日志返回时，建议 QA 如果看着堆栈懵逼，就去 `read_file` 具体的错误行附近代码。

### 4. 状态传递（Artifact 传递）
- 将各个阶段生成的方案 (`_solution.md`) 作为强约束，Coder 在编码时如果感觉偏离，可以回头翻看。

## 三、 本期落地实施计划

我们将对现有的 `coder_agent.py` 和 `test_agent.py` 进行如下增强：

1. **改造 `coder_agent.py` 的 System Prompt**：
   - 增加**规划期指导**：明确要求在写大规模代码前，先写依赖或者建立合理的目录结构。
   - 明确允许并鼓励使用 `pip install` 等命令准备沙箱环境。

2. **改造 `test_agent.py` 的策略**：
   - 强调**白盒测试**思想：先通过 `list_dir_tool` 和 `read_file_tool` 理解 Coder 写的接口协议，然后再下手写测试，而不是乱猜。
   - 强化 Assert 和依赖补齐：如果缺少 `pytest` 或其他包，自己装。

3. **异常处理优化**：
   - 在 `run_command_tool` 失败时，提示词中加入“如果你需要，请利用 `read_file_tool` 查看报错定位的具体文件代码”的强力暗示。

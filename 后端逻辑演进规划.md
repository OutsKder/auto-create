# 织界引擎 MVP：后端逻辑演进与架构规划

作为团队协作的一环，为了在**尽量不破坏现有前端视觉与交互（UI/UX）**的前提下，让「织界引擎」从一个“纯前端本地存储的 Demo”真正蜕变为“具备业务逻辑的引擎平台”，特制定本后端演进规划。

## 1. 演进原则

1. **前后端分离不阻断**：前端保留现有的 HTML/CSS 和基础 DOM 操作结构。我们将原本直接读写 `localStorage` 的逻辑，逐步替换为 `fetch` / `WebSocket` 调用，保持页面视觉稳定性。
2. **渐进式真实化**：
   - **Phase 1 (Mock API)**：搭建 Node.js 后端，用 API 吐出现在的假数据，完成前后端接口联调。
   - **Phase 2 (Real Engine)**：后端引入真实的状态机引擎和 LLM（大模型）调用，实现真正的 AI Agent 调度。
3. **贯彻核心理念**：后端架构必须体现「Pipeline 是骨架，Agent 是肌肉，人类是大脑」的设计初衷。

---

## 2. 后端核心模块设计 (Backend Architecture)

为支持任意需求输入并实现真实运转，后端需划分为三个核心域：

### 2.1 流程引擎服务 (Pipeline Engine)
**职责**：充当“骨架”，基于有限状态机（FSM）管理一条需求所处的流转阶段。
- 维护 `STAGE_DEFS` 规则（流转图、准入条件、出口条件）。
- 遇到 `requiresReview: true` 节点（如方案设计输出后、编码前），将状态挂起为 `waiting_review`，并中断 Agent 执行，等待前端发送的人工审核指令（`approve` / `reject`）。

### 2.2 智能体调度网关 (Agent Dispatcher)
**职责**：充当“肌肉”，封装对大模型的基础调用，并将上游节点的结构化结果作为下游的 Context (上下文) 传入。
- `AnalysisAgent`：接收原始需求文本 -> 输出结构化分析（JSON/Markdown）。
- `SolutionAgent`：接收分析结果 -> 输出技术方案架构。
- `CodingAgent`、`TestingAgent` 等。
- 提供流式（Server-Sent Events / WebSocket）输出能力，使前端能展示 Agent 的“沉浸式打字执行过程”与实时日志。

### 2.3 数据持久层 (Data Persistence)
**职责**：替代前端的 `localStorage`。MVP 阶段可选用轻量级的 SQLite 或直接 JSON 文件存储，以便于部署和演示。
- 存储 `Requirement` (需求元数据)、`StageResult` (各节点的 Agent 产物)、`AuditLog` (人工操作与引擎流转日志)。

---

## 3. 核心 API 契约初步设计 (API Contract)

前端的 `app.js` 将把现有的同步函数改造为异步请求，对接以下核心接口：

| 接口路线 | 方法 | 路径 | 核心作用 |
| :--- | :--- | :--- | :--- |
| **需求管理** | POST | `/api/requirements` | 接收前端表单，创建需求并在数据库初始化 Pipeline |
| | GET | `/api/requirements/:id` | 获取需求详情及当前所有阶段的产物状态 |
| **引擎驱动** | POST | `/api/pipeline/:id/trigger` | 触发某个阶段的 Agent 执行（支持 SSE 流式返回执行日志和内容） |
| **人工干预** | POST | `/api/pipeline/:id/approve` | 人脑介入：审核通过，状态机流转到下一节点 |
| | POST | `/api/pipeline/:id/reject` | 人脑介入：驳回，允许修改批注并要求 Agent 重新生成 |

---

## 4. 实施阶段拆解与协作分工 (Roadmap)

### 第一阶段：接口剥离与 Mock Server 联调（本周重点）
- **后端动作**：快速初始化一个 Node.js (Express/Koa) 工程。写死数据，提供上述 REST API。
- **前端动作**：改造 `app.js`。将 `saveState()` 和 `loadState()` 移除，将创建需求、点击“审核通过”等动作改成 `fetch` 请求。
- **目标**：跑通前后端数据交互，前端 UI 没有任何降级。

### 第二阶段：引擎状态机与流式响应开发
- **后端动作**：引入状态库，校验流转合法性（未 analysis 不能直接 solution）。实现 SSE 接口，让前端日志面板能一行行打印 `[Agent] 正在提取需求目标...`。
- **前端动作**：接收 SSE 流数据，实现打字机效果的前端渲染更新。

### 第三阶段：接入大模型 (Real LLM Integration)
- **后端动作**：在 Agent Dispatcher 中接入真实 LLM API（如 DeepSeek/OpenAI），拼装 Prompt：`"根据上游需求 ${context}，请直接输出测试用例..."`。
- **前端动作**：无感知。但用户将看到真正的“输入不同，输出不同”的动态效果。

---

## 5. 总结

通过上述规划，我们可以做到**前端视觉和交互继续独立优化（保留高品质呈现）**，同时后端逻辑逐渐由“假”变“真”。一旦后端接入真实大模型调度，我们在比赛中呈现的将不再是一个预设幻灯片，而是一个**具有业务深度、可直接商业落地的 AI 工作流中台**。

# 基于 AI 驱动的需求交付流程引擎

## 1. 项目概述

### 1.1 项目名称
需求交付流程引擎（AI-Driven Delivery Engine）

### 1.2 一句话定义
面向研发团队的 AI 全链路交付工作台，通过流程引擎串联 `需求 -> 方案 -> 编码 -> 测试 -> 评审 -> 交付`，实现多 Agent 协作执行、关键节点人工校验与全流程状态可视化。

### 1.3 项目背景
传统需求交付流程中，需求信息分散在聊天、文档、会议纪要、任务系统和代码仓库中，跨角色协同依赖大量人工串联与反复确认。现有 AI 工具往往只覆盖 Coding 环节，无法打通从需求理解到最终交付的完整链路，导致流程割裂、状态不透明、沟通成本高、质量难追踪。

### 1.4 项目目标
- 将需求交付流程标准化为可执行 Pipeline。
- 让 AI 从单点工具升级为贯穿全链路的执行主体。
- 在关键节点保留人工审核与纠偏能力，形成可控的人机协同闭环。
- 将需求状态、产出物、风险点和执行记录统一沉淀到一个可视化平台中。

### 1.5 核心理念
`Pipeline 是骨架，Agent 是肌肉，人类是大脑。`

---

## 2. 问题定义

### 2.1 当前流程的典型痛点
- 需求信息源分散，背景、目标、约束条件难以完整收集。
- 需求到方案、方案到开发、开发到测试之间需要大量人工同步。
- 不同角色对需求理解不一致，返工频繁。
- AI 只参与代码生成，前后环节仍然依赖人工流转。
- 缺少统一状态面板，项目 owner 很难快速判断当前卡点。
- 需求变更后，方案、代码、测试和交付材料无法自动联动更新。

### 2.2 目标用户的真实困境
- 产品经理：需求整理和跟进成本高，信息回收慢。
- 开发负责人：方案评估、任务拆解和代码交付上下文断裂。
- 测试负责人：测试依据不统一，验收口径不清晰。
- 项目 owner：全流程不可见，风险发现滞后。

### 2.3 本产品要解决的核心问题
如何构建一个既能自动推进需求交付流程，又允许人在关键节点进行审核和接管的 AI 流程引擎，使需求从输入到交付具备标准化、自动化、可控化和可视化能力。

## 3. 产品定位

### 3.1 产品定位
一个面向中小研发团队和创新项目团队的 AI 需求交付工作台。

### 3.2 产品边界
本产品不试图替代全部专业角色，而是作为一套流程中台和智能执行层，帮助团队减少重复沟通、缩短交付周期、提升交付透明度与质量稳定性。

### 3.3 核心价值
- 对团队：减少串联成本，缩短需求交付路径。
- 对管理者：提升流程透明度和交付可控性。
- 对执行者：降低上下文切换和重复文档劳动。
- 对比赛：体现完整链路、多 Agent 协作和人机协同的创新价值。

## 4. 目标用户与角色设计

### 4.1 核心角色
- 需求发起人：提交需求背景、目标和约束。
- 产品经理 / 项目 Owner：确认需求、审核方案、推进流程。
- 开发负责人：确认技术方案、审核代码实现。
- 测试负责人：确认测试策略与验收结果。
- AI Agent：在不同节点执行分析、生成、校验和总结任务。

### 4.2 角色诉求

| 角色 | 核心诉求 | 当前痛点 |
|------|----------|----------|
| 需求发起人 | 快速表达需求并获得可落地结果 | 不知道需求目前推进到哪一步 |
| 产品经理 | 统一需求材料，推动交付 | 各环节反复沟通、来回确认 |
| 开发负责人 | 快速获得明确方案和任务拆解 | 信息不完整、返工成本高 |
| 测试负责人 | 基于清晰验收标准开展测试 | 缺少统一输入和过程记录 |
| 项目 Owner | 看清状态、风险、进度 | 无法实时把握流程卡点 |

### 4.3 核心使用场景
- 场景一：从零开始处理一个新增功能需求。
- 场景二：需求发生变更，系统自动识别受影响的方案、代码和测试环节。
- 场景三：交付前需要统一查看需求链路中的关键产物和审批记录。

## 6. 产品愿景与设计原则

### 6.1 产品愿景
打造一个让需求交付像运行工作流一样稳定、透明、可复用的 AI 引擎平台。

### 6.2 设计原则
- 全链路优先：不能只做单点 AI 能力。
- 闭环优先：流程必须完整跑通。
- 可控优先：关键节点必须可人工接管。
- 透明优先：每一步状态、输入、输出、责任方都可追踪。
- 可扩展优先：后续可新增 Agent、节点和流程模板。

## 7. 方案总体设计

### 7.1 总体架构
产品由四层构成：

1. **流程层**：定义需求交付 Pipeline、节点顺序、状态流转和触发规则。
2. **Agent 层**：针对不同阶段配置专属 Agent，执行分析、生成、校验和总结。
3. **人工协同层**：在关键节点提供审核、修改、驳回、重跑能力。
4. **可视化层**：展示需求状态、执行日志、阶段产物、风险提示和交付结果。

### 7.2 核心流程
`需求录入 -> 需求理解 -> 方案生成 -> 人工确认 -> 编码执行 -> 测试生成与执行 -> 评审 -> 交付归档`

### 7.3 流程引擎职责
- 管理流程模板和节点配置。
- 决定不同节点的执行顺序和依赖关系。
- 管理阶段状态，如待处理、执行中、待审核、已通过、已驳回、已完成。
- 保存上下文和阶段产物，传递给后续 Agent。

### 7.4 Agent 编排职责
- 接收节点任务和上下文。
- 产出结构化结果供后续节点复用。
- 根据规则触发下一节点或提交人工审核。
- 在失败时提供重试或回退建议。

## 8. MVP 范围定义

### 8.1 MVP 目标
在有限时间内完成一个可演示、可答辩、可扩展的最小闭环版本，重点证明系统具备流程引擎能力，而不是堆叠功能。

### 8.2 推荐 MVP 闭环
建议优先实现以下链路：

`需求 -> 方案 -> 编码 -> 测试 -> 评审 -> 交付`

### 8.3 MVP 必做功能
- 新建需求并录入背景信息。
- 自动调用需求分析 Agent 生成需求摘要、关键问题和验收目标。
- 自动调用方案 Agent 生成技术方案草案。
- 在方案节点支持人工确认 / 编辑 / 驳回。
- 自动调用编码 Agent 生成代码草案或任务实现建议。
- 自动调用测试 Agent 生成测试用例和验收清单。
- 自动调用评审 Agent 生成评审摘要和风险提示。
- 展示完整流程状态、节点结果和执行日志。
- 支持至少一个关键节点的人工重跑和回退。

### 8.4 MVP 暂不做
- 复杂权限系统
- 多项目多租户管理
- 完整企业级代码仓库集成
- 复杂监控告警体系
- 高度自动化生产可用交付能力

## 9. 功能模块拆解

### 9.1 需求接入模块
功能目标：统一收集需求背景，形成标准化输入。

核心能力：
- 需求标题、背景、目标、约束录入
- 附件或补充材料接入
- 需求优先级、截止时间、负责人配置
- 自动生成需求摘要

### 9.2 流程编排模块
功能目标：定义和驱动需求在各阶段之间流转。

核心能力：
- 预置流程模板
- 节点顺序和依赖配置
- 节点状态管理
- 节点失败重试与回退

### 9.3 Agent 执行模块
功能目标：在不同阶段自动完成分析与生成任务。

核心能力：
- 需求分析 Agent
- 方案设计 Agent
- 编码 Agent
- 测试 Agent
- 评审 Agent
- 交付总结 Agent

### 9.4 人工审核模块
功能目标：在关键决策点提供人工把关能力。

核心能力：
- 审核通过
- 审核驳回
- 在线编辑
- 重跑节点
- 添加人工备注

### 9.5 状态展示模块
功能目标：让流程可见、可追踪、可复盘。

核心能力：
- 流程全景图
- 当前节点状态
- 阶段产出物预览
- 风险提示
- 执行日志
- 责任人和时间线展示

### 9.6 交付归档模块
功能目标：在流程结束后沉淀完整交付材料。

核心能力：
- 汇总需求、方案、代码、测试、评审结果
- 生成交付摘要
- 支持导出为文档或报告

## 10. Agent 体系设计

### 10.1 Agent 角色划分

| Agent | 输入 | 输出 | 作用 |
|------|------|------|------|
| 需求分析 Agent | 原始需求、背景材料 | 需求摘要、澄清问题、验收目标 | 将模糊需求结构化 |
| 方案设计 Agent | 结构化需求、技术约束 | 技术方案草案、模块拆解、风险点 | 生成可执行方案 |
| 编码 Agent | 已确认方案、任务拆解 | 代码草案、实现说明 | 输出研发实现建议 |
| 测试 Agent | 需求与代码上下文 | 测试用例、验收建议、风险项 | 保障测试覆盖 |
| 评审 Agent | 代码、测试、方案 | 评审意见、问题列表、通过建议 | 辅助质量评估 |
| 交付 Agent | 全流程产物 | 交付总结、沉淀文档 | 形成闭环输出 |

### 10.2 Agent 协作机制
- 每个 Agent 接收前一节点的结构化产物而非纯自然语言。
- Agent 输出统一采用结构化格式，便于流程引擎消费。
- 流程引擎负责管理上下文继承和节点触发。
- 高风险节点输出后进入人工审核状态，不直接自动放行。

### 10.3 Agent 设计亮点
- 多 Agent 分工明确，避免单一大模型承担所有角色。
- 将复杂交付任务拆为阶段性任务，提高可靠性。
- 可扩展新 Agent，如风险评估 Agent、排期 Agent、文档 Agent。

## 11. 人机协同设计

### 11.1 为什么必须加入人工校验
- 需求和方案存在业务理解偏差风险。
- 代码与测试结果涉及质量和责任边界。
- 全自动流程在真实业务中可解释性和可信度不足。

### 11.2 建议设置人工校验的关键节点
- 需求理解确认
- 技术方案确认
- 代码提交前确认
- 测试结果确认
- 最终交付确认

### 11.3 人工交互动作
- 通过：进入下一节点
- 驳回：退回上一节点或重跑当前节点
- 编辑：直接修改节点产物
- 接管：由人工继续完成当前步骤
- 备注：为后续 Agent 提供补充上下文

### 11.4 人机协同原则
- 低风险步骤自动推进
- 高风险步骤人工确认
- 节点失败时给出原因与修复建议
- 所有人工操作保留日志，便于追踪和复盘

## 13. 核心用户流程

### 13.1 主流程
1. 用户提交需求并填写背景、目标、限制条件。
2. 需求分析 Agent 自动生成需求摘要、澄清点和验收目标。
3. 方案设计 Agent 基于需求输出技术方案和任务拆解。
4. 项目 Owner 或开发负责人审核方案，可通过、修改或驳回。
5. 编码 Agent 基于通过的方案输出实现结果。
6. 测试 Agent 自动生成测试用例和验收建议。
7. 评审 Agent 汇总代码和测试结果，输出评审意见。
8. 关键负责人完成最终确认，系统生成交付报告并归档。

### 13.2 异常流程
- 如果需求不完整，需求分析 Agent 触发补充材料请求。
- 如果方案审核未通过，退回方案节点重生成。
- 如果测试未通过，退回编码节点修复。
- 如果评审存在高风险项，暂停交付并要求人工处理。

## 15. 竞争与差异化思考

### 15.1 相对传统项目管理工具的差异
- 不是静态记录工具，而是主动驱动流程的执行系统。
- 不是靠人手动推进，而是 AI 根据规则自动推进。
- 不是只有任务状态，而是同时沉淀上下文、产出物和决策记录。

### 15.2 相对单点 AI 编码工具的差异
- 不只覆盖编码，而是覆盖需求到交付的完整链路。
- 强调多 Agent 协作，而不是单一模型对话。
- 强调关键节点人工审核，提升可控性和可信度。

### 15.3 本方案的创新点
- 用流程引擎串联多个 Agent，形成结构化协作体系。
- 将人工审核节点设计为流程内生能力，而非外挂操作。
- 用状态可视化和阶段产物追踪提升需求交付透明度。

## 16. 技术实现建议

### 16.1 建议技术方向
- 前端：用于展示需求列表、流程状态和节点操作界面
- 后端：用于流程编排、任务调度、状态管理和日志存储
- 大模型 / Agent 层：用于执行需求分析、方案生成、代码与测试生成等任务
- 数据层：用于保存需求、节点状态、阶段产物和执行记录

### 16.2 系统关键能力
- 工作流编排
- 结构化上下文管理
- Agent 调用与结果回传
- 人工审核和回退控制
- 状态同步与日志记录

### 16.3 比赛实现策略
- 优先做单流程模板，不做复杂通用化配置。
- 优先做可演示的闭环，不追求生产级稳定性。
- 可采用模拟仓库、模拟测试结果或轻量集成方式完成 demo。

## 18. 风险与应对

### 18.1 产品风险
- 流程过长导致 MVP 无法按时完成
- Agent 输出不稳定影响演示效果
- 节点设计过于复杂，用户难理解

### 18.2 应对策略
- 优先保证最小闭环跑通
- 关键节点引入人工确认减少失控风险
- 使用固定模板和限定场景增强稳定性
- 用结构化输出约束 Agent 结果

## 19. 版本规划建议

### V2：增强版本
- 支持需求变更联动
- 支持多个流程模板

### V3：产品化版本
- 接入真实代码仓库和测试环境
- 支持多项目协同和权限体系
- 支持流程运营分析和组织级沉淀

## 20. 答辩表达建议

### 20.1 一句话价值主张
我们不是在做一个会写代码的 AI，而是在做一个能驱动需求交付全流程的 AI 流程引擎。

### 20.2 答辩重点
- 为什么传统流程低效
- 为什么单点 AI 不够
- 为什么需要 Pipeline + 多 Agent + 人工校验
- 你们的系统如何真正跑通一个闭环
- 这个方案未来如何接入真实团队工作流

### 20.3 评委最容易被打动的点
- 产品视角完整，不只是技术堆叠
- 流程设计真实，贴近研发交付场景
- 人机协同清晰，避免“纯自动化幻想”
- Demo 连贯，结果可见，状态透明

---

## 21. 附录：一句话总结
本项目旨在构建一个面向研发团队的 AI 驱动需求交付流程引擎，以流程引擎为骨架、以多 Agent 协作为执行核心、以人工审核为控制机制，最终实现需求从提出到交付的标准化、自动化、可控化和可视化。

### 🛡️ 项目上线评审 Checklist (Release Gatekeeper)

#### 1. 功能完备性与边界测试 (Functional & Edge Cases)
*我们不仅要测“能跑”，还要测“崩溃点”。*

- [ ] **空值与非法输入校验**：前端输入框禁止提交纯空格；后端接口必须校验字段长度（防止超长字符串导致数据库溢出）。
- [ ] **特殊字符渲染安全**：输入包含 `<script>` 或 HTML 标签的任务描述，在渲染时必须转义，严禁直接 innerHTML 渲染（防止 XSS 攻击）。
- [ ] **状态流转正确性**：点击“完成”后，任务列表是否按预期排序？再次点击是否恢复？删除操作是否有二次确认或软删除机制？
- [ ] **断网/弱网处理**：在添加任务时网络中断，是否有友好提示？避免用户误以为已保存实际失败。
- [ ] **刷新持久化**：页面 F5 刷新后，待办列表状态（已完成/未完成）是否正确保持？（检查 Session/Token 或 DB 存储策略）。

#### 2. 安全性 (Security) 🔴 **红线项**
*这是不可逾越的底线，尤其是涉及用户数据的 Web App。*

- [ ] **XSS 漏洞扫描**：使用工具对输入框进行 XSS payload 测试，确保无脚本执行风险。
- [ ] **身份认证与授权 (IAM)**：
    - [ ] *强制确认*：如果是团队使用，是否实现了用户登录体系？还是仅依赖 IP 隔离？（当前需求未明确，需补充说明）
    - [ ] 不同用户的数据是否在物理或逻辑层面完全隔离？A 用户能否通过改 ID 看到 B 用户的任务？
- [ ] **CSRF 防护**：POST/DELETE 请求是否携带 Token 验证？
- [ ] **敏感信息泄露**：API 响应中是否包含了无关的系统路径、堆栈信息或多余的用户隐私数据？
- [ ] **传输加密**：全站是否强制 HTTPS？

#### 3. 性能与稳定性 (Performance & Stability)
*简单的功能也扛不住高并发或大数据量。*

- [ ] **首屏加载时间 (FCP)**：任务列表页 FCP 必须在 **1.5 秒** 以内。
- [ ] **大数据量分页/虚拟滚动**：当单个用户任务超过 **500 条** 时，页面是否卡顿？是否实施了分页或懒加载策略？
- [ ] **接口响应延迟**：核心接口（Add/Complete/Delete）P99 延迟不得高于 **200ms**。
- [ ] **压力测试**：模拟 10 QPS 并发下的服务响应情况，CPU/内存占用率是否正常？

#### 4. 可观测性与监控 (Observability & Monitoring)
*上线后不是结束，是运维的开始。不知道出错了就是最大的错误。*

- [ ] **业务日志埋点**：关键操作（创建、完成、删除）是否有审计日志（Who, When, What, Old_Value, New_Value）？
- [ ] **异常报警**：服务器 Error Rate 超过 **1%** 或 接口延迟突增，是否配置了告警（邮件/钉钉/Slack）通知 On-call 人员？
- [ ] **健康检查接口**：`/healthz` 或 `/ping` 接口是否可用，并返回真实的 DB/Cache 连接状态？
- [ ] **前端错误上报**：JS Runtime Error 是否接入 Sentry 或类似监控平台？

#### 5. 部署与回滚 (DevOps & Rollback)
*如果上线搞挂了，你有后悔药吗？*

- [ ] **蓝绿/灰度发布方案**：是否支持先切流 5% 流量观察，有问题立即切断？
- [ ] **数据库变更兼容**：如果有新增字段或索引，是否具备向下兼容性？旧版本应用是否仍能正常运行？
- [ ] **配置管理**：生产环境的 Config (如 DB 地址、Secret Key) 是否与开发环境彻底隔离？严禁硬编码敏感信息！
- [ ] **一键回滚脚本**：验证过回滚流程，能在 **5 分钟** 内恢复至上一个稳定版本。

### ⚠️ Owner 特别风险提示 (Critical Gap Analysis)

作为技术 Owner，我必须指出当前需求文档中存在的**严重缺失**，请在上线前补充设计文档并通过评审：

1.  **数据所有权不明确**：需求提到“团队”，但未定义权限模型。是“公共白板”还是“私人空间”？**如果未做鉴权直接上线，视为重大安全缺陷。**
2.  **数据持久化策略缺失**：如果用户清除了浏览器缓存，本地存储的待办事项是否会永久丢失？是否需要后端兜底？
3.  **软删除 vs 硬删除**：直接物理删除（Hard Delete）是不允许的。必须采用软删除（标记 IsDeleted），以便后续用户找回误删的重要工作记录。

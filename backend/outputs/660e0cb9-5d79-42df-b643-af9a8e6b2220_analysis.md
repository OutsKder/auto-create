### 一、商业价值与产品愿景 (Vision & Value)

- **真伪需求研判**
  - **表象需求**：在现有列表中增加“单价”和“库存”两个字段及显示逻辑。
  - **核心痛点**：当前系统处于“资产黑盒”状态。后勤部门面临盘点即混乱的风险，且无法通过数据驱动采购决策。真正的痛点并非“录入信息”，而是**“资产实时可见性缺失”**导致的资金占用浪费（积压）或业务中断风险（缺货）。
  - **替代方案评估**：若仅做静态列表，本质上只是将 Excel 搬上 Web，未形成闭环。但考虑到周末盘点的紧迫性，MVP 阶段优先保证“可记录、可展示”，后续需迭代为“可流转、可审批”。

- **业务核心收益**
  - **降本增效**：建立库存基线，预期减少 15%-20% 的非必要紧急采购成本；缩短盘点核对时间。
  - **合规风控**：满足财务审计对固定资产/耗材估值的数字化追溯要求，规避资产流失风险。
  - **体验提升**：通过红色 Badge 显性化预警，降低人工记忆负荷，将被动响应转变为主动干预。

- **北极星指标 (North Star Metric)**
  - **库存数据准确率** = (账实相符的 SKU 数量 / 总盘点 SKU 数量) × 100%
  - *定义*：该功能上线并运行一个周期后，后台数据与实际物理库存的一致性比例。单纯看“新增条目数”无意义。

### 二、用户画像与核心场景 (Personas & Scenarios)

- **利益相关者**
  - **后勤管理员 (Admin)**：核心录入与维护者。诉求：操作极简，避免繁琐流程影响盘点效率，能快速识别异常。
  - **财务人员 (Finance)**：资产估值审核者。诉求：单价准确，数据可导出或对接报表，支持审计追踪。
  - **部门申领人 (Requester)**：间接使用者。诉求：能看到是否有货，避免提交无效申请导致挫败感。

- **典型用户故事 (User Story)**
  1.  **作为** 后勤管理员，**我希望** 在添加新物品时能直接录入采购价和初始库存，**以便于** 建立准确的资产台账，无需二次补录。
  2.  **作为** 后勤管理员，**我希望** 当某物品库存低于 5 件时在列表高亮警示，**以便于** 优先处理该类物品的补购，防止业务断供。
  3.  **作为** 财务审计员，**我希望** 系统能展示所有物品的预估总价值（单价×库存），**以便于** 快速完成季度资产估值核算。

### 三、MVP 核心功能拆解边界 (Scope & Features)

- **核心功能必做 (Must-haves)**
  - **数据模型迁移**：后端 `Item` 表结构变更，确保 `price` 使用 Decimal 类型而非 Float（避免精度丢失），`stock` 为非负整数。
  - **表单校验增强**：前端输入框必须限制 `price > 0` 且保留两位小数，`stock ≥ 0`。禁止空值提交破坏数据完整性。
  - **动态渲染逻辑**：列表项前端判断逻辑 `if (stock < 5) render Badge('bg-danger', '库存告急')`，且该状态需实时生效。
  - **存量数据处理**：针对已存在的旧数据，提供默认值填充策略（如单价默认为 0 或标记为“待确认”），避免查询报错。

- **伪需求与延期池 (Won't-haves)**
  - **库存流水日志**：本期不记录库存变动历史（谁、何时、减了多少），仅展示当前快照。
  - **低库存自动采购单**：发现 `<5` 时不自动触发采购流程，仅限视觉提醒。
  - **多仓库/多地点管理**：暂不支持分仓库存逻辑，全量合并计算。
  - **图片上传/条码扫描**：维持纯文本录入，降低开发复杂度以赶上周末盘点。

- **非功能性需求**
  - **数据一致性**：在高并发录入（如多人同时盘点）场景下，`stock` 字段需保证最终一致性，虽不强制乐观锁，但需有防超卖基础校验。
  - **性能底线**：列表加载接口响应时间不超过 500ms，即使数据量增长至 10,000+ 条，也不应阻塞后台管理页面。
  - **安全性**：价格与库存属于敏感商业数据，需控制 API 权限，仅授权角色可读写。

### 四、潜在风险与可行性预警 (Risks & Feasibility)

- **业务运营卡点**
  - **数据维护惰性**：增加字段后，录入成本微增，员工可能随意填写“99999”或"0"以跳过思考，导致预警失效。**对策**：需配套简单的数据质量清洗规则或抽查机制。
  - **阈值僵化**：硬编码 `< 5` 作为警戒线缺乏灵活性。不同物品（如 A4 纸 vs 笔记本电脑）的安全库存水位完全不同。**对策**：预留字段扩展空间，下期迭代支持配置化阈值。
  - **盘点时效冲突**：本周末急需上线，可能导致测试不充分。若上线后出现 Bug 回滚，将直接影响盘点工作。

- **技术预警**
  - **精度陷阱**：若后端语言默认使用浮点数存储价格（如 JS Number 或 Java Double），会导致金额计算误差。**严禁**使用浮点型存储货币。
  - **遗留代码耦合**：在现有“列表骨架”上叠加新功能，需警惕旧版前端模板未适配新字段导致的样式崩坏（UI Regression）。
  - **弱网同步**：若盘点现场网络不佳，纯在线模式可能导致数据丢失，需考虑本地缓存或离线容错机制（视具体架构而定）。

### 五、最优先级澄清清单 (Top Clarification Questions)

1.  **关于“单价”的时效性**：录入的 `price` 是指“当前账面成本价”还是“最新采购成交价”？若同一物品多次采购价格不同，是否覆盖？这决定了数据模型是否需要关联采购记录。
2.  **关于“库存扣减”逻辑**：目前的库存是“手工修正式”（每次盘点重填）还是“事务式”（入库 +1，领用 -1）？若是前者，本需求仅为快照展示；若是后者，缺少了领用模块，库存将迅速失准。

# 待办事项 Web App 技术架构方案草案 (v1.0)

## 1. 总体架构设计 (Architecture Overview)

鉴于需求初期规模较小，但考虑到未来可能扩展为团队任务管理，建议采用 **前后端分离的模块化单体架构 (Modular Monolith)**。这种模式在开发效率和代码解耦之间取得了最佳平衡，且方便后续拆分为微服务。

*   **前端架构**：采用现代 SPA 框架（如 React / Vue 3），配合 TypeScript。支持 PWA (Progressive Web App)，实现离线缓存和本地优先策略，保证在无网环境下仍可操作。
*   **后端架构**：Node.js (NestJS) 或 Go (Gin/Zap)。采用分层架构（Controller -> Service -> Repository）。
*   **数据存储**：PostgreSQL（关系型数据，确保 ACID，方便未来做复杂查询和事务）。
*   **部署方式**：Docker 容器化 + Kubernetes（预留弹性伸缩能力）。

### 核心原则
1.  **Stateless**：API 无状态，便于水平扩展。
2.  **Optimistic UI**：前端优化感知，用户点击即反馈，后台异步落库。
3.  **Extensibility**：数据库字段设计预留扩展位（如 priority, category_id），避免频繁改表结构。

| 模块名称 | 职责描述 | 关键技术点 |

| **Audit Log (审计日志)** | *(扩展预留)* 记录关键操作，便于排查问题。 | Write-Ahead Log |

## 3. API 设计思路 (API Design Strategy)

遵循 RESTful 规范，统一返回格式，使用版本控制 (`/api/v1/...`) 以防未来迭代破坏旧接口。

### 3.1 核心资源定义
*   `Task`: 待办任务实体
*   `User`: 用户实体

### 3.2 关键接口定义

| 方法 | URI | 描述 | 请求体示例 | 响应码示例 |
| :--- | :--- | :--- | :--- | :--- |
| **POST** | `/api/v1/tasks` | 创建新任务 | `{ "title": "写周报", "priority": 1 }` | `201 Created` |
| **GET** | `/api/v1/tasks` | 获取任务列表 | Query Params: `?status=pending&page=1&limit=20` | `200 OK` |
| **PATCH** | `/api/v1/tasks/:id` | 更新任务状态/内容 | `{ "is_completed": true }` | `200 OK` |
| **DELETE** | `/api/v1/tasks/:id` | 删除任务 | - | `204 No Content` |
| **PUT** | `/api/v1/auth/login` | 用户登录 | `{ "email": "...", "password": "..." }` | `200 OK` (Returns Token) |

### 3.3 设计亮点
1.  **幂等性 (Idempotency)**：对于创建接口，客户端生成唯一 `client_id` 防止网络重试导致重复插入任务。
2.  **乐观并发控制**：Update 接口携带 `updated_at` 时间戳或版本号，防止并发修改导致的数据覆盖。
3.  **批量操作**：虽然当前需求是单条，但预留 `POST /batch/delete` 接口，为未来支持多选做准备。

## 4. 数据库核心表设计 (Database Schema)

选择 **PostgreSQL**，利用其强大的类型支持。

### 4.1 用户表 (`users`)
```sql
CREATE TABLE users (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(), -- 使用 UUID 防遍历猜测
    email       VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at  TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    last_login_at TIMESTAMPTZ
);
CREATE INDEX idx_users_email ON users(email);
```

### 4.2 任务表 (`tasks`) - 核心表
```sql
CREATE TABLE tasks (
    id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id        UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, -- 关联用户，支持多租户
    
    content        TEXT NOT NULL,              -- 任务内容
    is_completed   BOOLEAN DEFAULT FALSE,      -- 是否完成
    completed_at   TIMESTAMPTZ,                -- 完成时间（用于统计）
    
    priority       INT DEFAULT 0,              -- 优先级 (-1:低，0:中，1:高)，为排序和过滤预留
    due_date       TIMESTAMPTZ,                -- 截止时间（预留扩展字段）
    
    is_deleted     BOOLEAN DEFAULT FALSE,      -- 软删除标记，允许数据恢复
    created_at     TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    updated_at     TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    
    version        INT DEFAULT 1               -- 乐观锁版本号
);

-- 复合索引：支持按用户筛选并按时间/优先级排序，极大提升列表加载速度
CREATE INDEX idx_tasks_user_status ON tasks(user_id, is_completed, created_at DESC); 
```

## 1. 总体设计愿景 (Vision & Overview)

虽然当前需求仅为“简单的个人待办记录”，但作为架构师，我必须确保系统具备**低耦合、高内聚**的特性。我们需要构建一个在 MVP（最小可行性产品）阶段足够轻量，但在未来面对**团队协作、多端同步、数据权限、高并发读写**场景下能够平滑演进的系统。

本方案采用 **前后端分离** 架构，核心关注点在于：
1.  **响应速度**：用户操作即时反馈（Optimistic UI）。
2.  **数据安全**：基于 RBAC 或 JWT 的用户隔离。
3.  **可维护性**：清晰的领域边界与接口规范。

## 2. 模块拆解 (Module Breakdown)

我们将系统划分为四个逻辑层级，确保关注点分离 (SoC)。

## 3. API 设计思路 (API Design Strategy)

遵循 **RESTful** 风格规范，资源导向设计。为了便于未来扩展，建议引入 API Versioning。

### 3.1 基础规范
*   **Method**: GET, POST, PUT, PATCH, DELETE
*   **Format**: JSON
*   **Auth**: Bearer Token (JWT) in Header
*   **Pagination**: 统一支持分页查询 (`page`, `page_size`)

### 3.2 核心接口定义

#### 3.2.1 任务管理 (Task Resource)

| 方法 | URL 路径 | 描述 | 关键参数/Body | 返回示例 |
| :--- | :--- | :--- | :--- | :--- |
| **POST** | `/api/v1/tasks` | **新建任务** | `{ "content": "...", "priority": 1 }` | `201 Created` + `task_id` |
| **GET** | `/api/v1/tasks` | **获取任务列表** | Query: `status=active|completed`, `date=` | `200 OK` + `[task_list]` (含总数) |
| **PATCH** | `/api/v1/tasks/{id}` | **更新任务状态** (如标记完成) | `{ "status": "completed" }` | `200 OK` + `updated_task` |
| **DELETE**| `/api/v1/tasks/{id}` | **删除任务** | N/A | `204 No Content` (建议使用软删除，见难点) |

#### 3.2.2 鉴权 (Auth Resource)

| 方法 | URL 路径 | 描述 |
| :--- | :--- | :--- |
| **POST** | `/api/v1/auth/login` | 用户登录，换取 Access/Refresh Token |
| **POST** | `/api/v1/auth/register` | 新用户注册 |

### 3.3 错误码设计
不要只返回 HTTP 状态码，需配合自定义 Error Code：
*   `4000`: 参数校验失败
*   `4010`: 未授权 (Token 过期)
*   `4030`: 禁止访问 (试图修改他人的任务)
*   `5000`: 服务器内部错误

## 4. 数据库核心表设计 (Database Schema Design)

考虑到未来的扩展性（如分类、标签、团队协作），我们在当前简单需求的基础上预留字段。采用 **MySQL/PostgreSQL**。

### 4.1 用户表 (Users)
存储账户安全信息。

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    username VARCHAR(50) UNIQUE NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL, -- 加密存储
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_login_at TIMESTAMP NULL,
    is_active BOOLEAN DEFAULT TRUE,
    INDEX idx_email (email)
);
```

### 4.2 待办任务表 (Todos)
核心业务表。**注意：所有操作必须带上 user_id 进行数据隔离。**

```sql
CREATE TABLE todos (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,              -- 数据隔离键
    title VARCHAR(255) NOT NULL,          -- 任务标题
    description TEXT,                     -- 详细描述 (可选)
    status TINYINT DEFAULT 0,             -- 0:未开始，1:进行中，2:已完成
    priority TINYINT DEFAULT 1,           -- 优先级 (用于排序)
    due_date DATETIME,                    -- 截止时间 (可选)
    is_deleted TINYINT DEFAULT 0,         -- 软删除标记 (为数据恢复留出可能)
    deleted_at DATETIME,                  -- 删除时间
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    
    CONSTRAINT fk_todos_user FOREIGN KEY (user_id) REFERENCES users(id),
    
    -- 索引优化：提升按用户查询及筛选性能
    INDEX idx_user_status (user_id, status, is_deleted),
    INDEX idx_user_due_date (user_id, due_date)
);
```

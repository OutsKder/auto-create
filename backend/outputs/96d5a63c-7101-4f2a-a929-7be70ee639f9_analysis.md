### 一、商业价值与产品愿景 (Vision & Value)

- **真伪需求研判**
    - **核心痛点剖析**：用户表面需要“待办工具”，实则痛点在于**信息碎片化导致的认知过载**与**任务优先级管理失效**。市场上已有成熟竞品（如 Microsoft To Do, Notion, TickTick），若仅为通用功能，自研的 ROI（投资回报率）极低。
    - **替代方案评估**：除非该工具能与内部现有系统（如 OA、IM、CRM）打通实现数据自动流转，否则员工倾向于使用手机自带备忘录或第三方应用。单纯开发一个独立 Web 端可能沦为“僵尸系统”。
    - **结论**：仅当定位为**企业内部轻量级流程入口**或**特定业务场景闭环**时，真实需求成立；否则为伪需求。

- **业务核心收益**
    - **效率提升**：通过标准化任务记录，减少口头沟通成本，预计降低 10%-15% 的因遗漏导致的返工成本。
    - **过程资产沉淀**：将隐性的个人工作计划转化为显性的数字化记录，便于后续复盘与绩效追溯。
    - **合规与安全**：若涉及敏感业务数据，内网部署可规避使用第三方公有云工具的数据泄露风险。

- **北极星指标 (North Star Metric)**
    - **周活跃完成者比率 (Weekly Active Completers / WAU)**：定义为“每周至少标记 3 个任务为完成的独立用户数 / 总活跃用户数”。
    - *理由*：单纯的“创建”不代表生产力，“完成”才代表价值交付。关注完成率而非使用频次更能衡量效率提升的真实性。

### 二、用户画像与核心场景 (Personas & Scenarios)

- **利益相关者**
    - **执行者（普通员工）**：核心诉求是**低摩擦记录**与**防遗忘提醒**。厌恶复杂配置，追求快速完成任务状态更新。
    - **管理者（团队 Leader）**：核心诉求是**进度可见性**与**风险预警**。希望了解团队负载分布，但不过度干涉个人隐私。
    - **运维管理员（IT/Admin）**：核心诉求是**数据安全**与**账号权限管控**。确保系统稳定，无非法访问风险。

- **典型用户故事 (User Story)**
    1.  作为 **执行者**，我希望 **在浏览器侧边栏或快捷键中一键唤起输入框**，以便于 **打断工作流不超过 5 秒地捕获临时灵感或紧急任务**。
    2.  作为 **执行者**，我希望 **任务完成后有明确的视觉反馈（如划线、归档）并收到次晨推送**，以便于 **建立正向反馈循环并防止重要事项被覆盖**。
    3.  作为 **管理者**，我希望 **在不打扰下属的前提下查看团队今日高优先级任务列表**，以便于 **识别潜在的资源瓶颈或延期风险**。

### 三、MVP 核心功能拆解边界 (Scope & Features)

- **核心功能必做 (Must-haves)**
    - **极速录入与编辑**：支持纯文本快速添加，支持回车换行批量录入，支持修改内容。
    - **状态流转机制**：待办 → 进行中 → 已完成 → 已删除（需包含回收站逻辑，支持误删恢复）。
    - **基础提醒服务**：基于时间点的站内信或邮件提醒（解决“漏掉工作”的核心背景）。
    - **数据持久化与同步**：多标签页同步，刷新不丢失数据，保证本地存储与服务端同步的一致性。
    - **身份认证**：必须与企业统一认证系统（SSO/LDAP）对接，禁止开放注册。

- **伪需求与延期池 (Won't-haves)**
    - **复杂标签与分类系统**：MVP 阶段严禁引入多级文件夹或自定义标签，这会极大增加学习成本。
    - **移动端 App**：明确限定为 Web App，优先利用响应式布局适配移动端浏览器，暂不开发原生 App。
    - **团队协作看板/甘特图**：这属于项目管理范畴，非个人待办工具的 MVP 范围，会导致项目严重蔓延。
    - **AI 智能排序/预测**：初期数据量不足，算法无意义，且引入不必要技术复杂度。

- **非功能性需求**
    - **性能要求**：首屏加载时间 < 1 秒，任务列表渲染延迟 < 200ms，避免卡顿影响心流。
    - **安全性**：所有接口 HTTPS 加密，数据库字段脱敏（若含敏感信息），操作日志全留存以备审计。
    - **兼容性**：必须兼容公司标配浏览器版本（如 Chrome, Edge 最新版本），不考虑老旧 IE。

### 四、潜在风险与可行性预警 (Risks & Feasibility)

- **业务运营卡点**
    - **习惯迁移阻力**：员工已形成固定记录习惯（如纸质、微信笔记），强行切换工具若无强制考核或深度集成，存活率极低。
    - **数据孤岛效应**：若待办事项无法关联到具体业务单据（如订单号、工单 ID），任务将流于形式，无法追踪实际产出。
    - **维护成本倒挂**：内部自研小工具长期维护的人力成本往往高于购买 SaaS 服务订阅费。

- **技术预警**
    - **弱网环境同步冲突**：用户在网络波动下频繁操作导致的数据版本冲突（Optimistic UI vs Server State），需设计合理的冲突解决策略（最后写入获胜或人工合并）。
    - **浏览器关闭后数据丢失**：前端存储（LocalStorage）可靠性差，必须确保每一次写操作异步即时同步至后端数据库。
    - **跨域与沙盒限制**：Web 端在嵌入内网门户时可能面临 Cookie 共享与会话保持的技术坑点。

### 五、最优先级澄清清单 (Top Clarification Questions)

为防止资源浪费，需向业务方发起以下尖锐质询：

1.  **差异化生存论据**：既然市面上存在大量免费且优秀的待办工具，我们**自研的核心护城河是什么**？是为了数据安全，还是为了与内部业务流程强绑定？如果是前者，是否有必要投入研发周期？
2.  **“重要工作”的定义**：背景中提到“经常遗漏重要工作”，请问**业务层面如何定义“重要”**？系统是否需要提供优先级字段？如果用户只标记不完成，系统是否具备升级催办的能力？
3.  **单一用户 vs 团队协同**：目标是“个人效率”还是“团队透明”？这两个方向的产品逻辑截然不同。如果是团队，是否意味着需要**上级审批**或**任务分派**功能？这将直接决定架构设计。
4.  **退出机制**：如果该项目上线 3 个月后，日活用户低于 20%，业务方是否接受**下线服务**并将流量引导至第三方成熟工具？**失败的成本谁来承担**？
5.  **集成边界**：该待办事项是否允许从外部来源（如邮件、会议日程）自动抓取生成

# SDET 完整闭环落地方案

## 1. 目标

本方案定义测试生成 Agent 的完整闭环：

1. 接收结构化输入：`code_diff`、`requirement_structured`、`design`、`codebase_context`
2. 先生成结构化 `TestBundle`
3. 在隔离工作区内应用代码变更
4. 将测试代码物化为真实测试文件
5. 在 Docker 或本地 Runner 中执行测试命令
6. 将执行结果回填为 `sandbox_result`
7. 在失败场景下回灌日志，支持二次分析和修复

目标不是“能跑一次测试”，而是让测试生成、文件落盘、隔离执行、失败分析形成可重复的工程闭环。

---

## 2. 总体分层

整个闭环建议拆成 5 层：

### 2.1 测试规划层

职责：

- 读取 `code_diff`、`requirement_structured.acceptance_criteria`、`design`
- 生成 `TestBundle.test_plan`
- 生成 `TestBundle.test_files`
- 生成 `TestBundle.test_code`
- 生成 `TestBundle.runner_commands`

说明：

- 该层只负责“生成计划和建议”，不直接改文件、不直接执行命令。
- 该层的输出必须是结构化数据，不能只返回自然语言。

### 2.2 工作区物化层

职责：

- 以原代码库名为目录名，在 `backend/agent/workspace` 下创建临时工作区
- 将原仓库复制到该临时工作区
- 把 `code_diff` 应用到临时工作区
- 把 `test_code` 写入 `test_files` 指定路径
- 确保原始仓库不被污染

说明：

- 这是完整闭环的关键层。
- 后续所有执行都只能针对临时工作区，不允许直接操作原仓库。

### 2.3 容器执行层

职责：

- 将临时工作区挂载到容器内 `/workspace`
- 执行测试命令
- 捕获 `stdout`、`stderr`、退出码和完整日志
- 支持网络禁用、CPU/内存限制、进程数限制、只读挂载

说明：

- Docker 执行是优先方案。
- 如果 Docker 不可用，可回退到本地 Runner，但要保留相同的结果格式。

### 2.4 结果分析层

职责：

- 接收 `SandboxResult`
- 判断成功、失败、超时、依赖缺失、断言失败、语法错误等类型
- 将失败日志整理成可回灌的诊断输入

说明：

- 失败日志不要直接丢给人看，要先结构化分类。
- 结果分析层应该输出“下一步怎么修”，而不仅是“哪里错了”。

### 2.5 编排层

职责：

- 串联上述各层
- 控制重试次数
- 控制失败熔断
- 决定是否进入下一轮 CodeGen 或人工确认

说明：

- 编排层不应该混进测试生成器内部。
- `SDET` 只负责编排输入到输出的单次闭环，真正的循环控制应由 workflow/pipeline 负责。

---

## 3. 当前实现边界

### 3.1 已有能力

当前代码库已经具备以下能力：

- 结构化模型：`TestBundle`、`TestPlanItem`、`TestFile`、`SandboxResult`
- 测试生成模板：严格 JSON 输出约束
- `Patcher`：支持 Search/Replace 形式补丁应用
- `WorkspaceManager`：负责在 `backend/agent/workspace/<原代码库名>` 下创建临时工作区
- `Runner`：支持本地执行和 Docker 执行
- `SDET`：可在临时工作区里应用 `code_diff`、物化测试文件并执行命令
- 集成测试：已验证基础闭环和隔离工作区闭环

### 3.2 仍然需要加强的点

当前仍需持续完善：

- Docker 真实运行环境验证
- 失败日志回灌后的自动修复策略
- 更完整的资源约束策略
- 更细粒度的测试文件物化规则
- 失败分类和重试上限控制

---

## 4. 工作区规则

### 4.1 工作区目录规则

临时工作区必须统一放在：

`backend/agent/workspace/<原代码库名>`

例如：

- 原仓库是 `frontend`
- 临时工作区就是 `backend/agent/workspace/frontend`

### 4.2 工作区生命周期

1. 创建前检查是否存在同名工作区
2. 若存在则先清理旧目录
3. 复制原仓库到工作区
4. 在工作区中应用补丁和写入测试文件
5. 执行测试
6. 测试结束后清理工作区

### 4.3 工作区安全要求

- 原仓库只作为源数据，不能被直接修改
- 补丁只能应用在临时工作区
- 测试文件只能写到临时工作区
- 路径解析必须做越界检查，防止写出工作区外

---

## 5. 补丁应用规则

### 5.1 输入形式

`code_diff` 中的每个补丁必须包含：

- `file_path`
- `change_type`
- `patch_format`
- `patch`
- `reason`
- `risk_level`

### 5.2 应用规则

- 只允许应用到临时工作区
- 先检查目标文件是否存在
- 解析 Search/Replace block
- 若 search block 找不到，则直接失败
- 若目标文件已经包含 replace block，可视为幂等应用成功
- 单个补丁失败时，应返回可读错误信息

### 5.3 回放要求

补丁必须能被审计和回放：

- 原始补丁内容要保留
- 实际应用结果要保留
- 应能定位到哪个文件、哪一段替换失败

---

## 6. 测试文件物化规则

### 6.1 物化目标

`TestBundle` 中的 `test_code` 和 `test_files` 不能停留在建议层，必须写成真实文件。

### 6.2 物化策略

- 若 `test_files` 非空，则以第一个文件承载 `test_code`
- 若 `test_code` 存在但 `test_files` 为空，则默认写入 `tests/test_autogen.py`
- 其他测试文件可写为占位文件或后续拆分文件
- 自动生成的测试文件需要补充项目根路径导入支持

### 6.3 物化要求

- 测试路径必须是相对路径
- 不能写出工作区外
- 写入前必须创建父目录
- 写入后可立即交给 Runner 执行

---

## 7. Docker 执行规则

### 7.1 默认原则

Docker 是优先执行方式。目标不是“能跑”，而是“可控地跑”。

### 7.2 默认加固项

容器执行建议默认带上：

- `--network none`
- `--cpus 1.0` 或更低
- `--memory 1g` 或按需配置
- `--pids-limit 256`
- `--cap-drop ALL`
- `--security-opt no-new-privileges`
- `--read-only`
- `--tmpfs /tmp`

### 7.3 挂载规则

- 临时工作区挂载到 `/workspace`
- 只允许容器读取/修改工作区中的文件
- 不允许访问宿主机其他路径

### 7.4 命令执行规则

- 按 `runner_commands` 顺序执行
- 任意一步失败立即停止
- 每一步都记录 `stdout`、`stderr`、退出码
- 所有命令和输出都要进入最终 `SandboxResult.logs`

### 7.5 失败日志回灌

Docker 执行失败后，必须将以下内容回传给分析层：

- 失败命令
- 退出码
- 标准输出
- 标准错误
- 涉及的测试文件路径
- 当前工作区的补丁摘要

---

## 8. 失败回灌与自愈

### 8.1 回灌输入

失败分析至少要包含：

- `SandboxResult.logs`
- `SandboxResult.exit_code`
- 失败的 `runner_commands`
- 当前 `code_diff`
- 当前 `test_files`
- 当前 `test_code`

### 8.2 分析目标

分析层要把失败归类为：

- 语法错误
- 导入错误
- 断言失败
- 依赖缺失
- 环境问题
- 超时
- 测试设计不足

### 8.3 自愈边界

- 只允许做最小范围修复
- 优先修测试，再考虑修代码
- 同类错误连续出现时应收敛并熔断
- 不允许为了通过测试而扩大无关修改

### 8.4 重试上限

建议最多重试 3 次：

1. 初次生成
2. 根据失败日志定点修复
3. 再次最小修复
4. 超过上限后进入人工确认

---

## 9. 推荐模块职责

### 9.1 WorkspaceManager

负责：

- 创建临时工作区
- 按原代码库名生成目录
- 清理临时工作区

### 9.2 Patcher

负责：

- 应用 Search/Replace 补丁
- 返回补丁应用结果
- 保证幂等和错误可读

### 9.3 SDET

负责：

- 生成 `TestBundle`
- 应用 `code_diff`
- 物化测试文件
- 调用 `Runner`
- 回填 `sandbox_result`

### 9.4 Runner

负责：

- 在本地或 Docker 中执行命令
- 返回结构化运行结果

### 9.5 Workflow / Pipeline

负责：

- 串起 CodeGen -> SDET -> Runner -> Triage
- 控制重试和熔断
- 不把所有逻辑堆在一个类里

---

## 10. 建议的执行顺序

推荐的执行顺序如下：

1. CodeGen 产出 `code_diff`
2. `WorkspaceManager` 创建临时工作区
3. `Patcher` 把 `code_diff` 应用到工作区
4. `SDET` 生成并物化测试文件
5. `Runner` 在 Docker 或本地环境中执行测试
6. `Triage` 分析失败日志
7. 必要时回到 CodeGen 进行最小修复
8. 成功后输出最终结果

---

## 11. 最终约束

这套闭环必须坚持以下原则：

- 原仓库只读
- 所有修改都先落临时工作区
- 测试代码必须真实落盘
- Docker 执行要尽量加固
- 失败结果必须可回灌、可分诊、可重试
- 超过重试上限必须停下并交给人工

---

## 12. 结论

完整闭环的本质不是“自动生成代码和测试”，而是：

- 让代码修改和测试执行都发生在隔离工作区
- 让所有输入输出都结构化
- 让失败能被分析并驱动下一轮最小修复
- 让流程可重复、可审计、可终止

当这四点都满足时，SDET 才真正从“测试建议生成器”升级为“可执行的测试闭环编排器”。

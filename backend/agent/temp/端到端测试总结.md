
### 第 2 阶段：自愈循环（多轮迭代）

```
┌─────────────────────────────────────┐
│    迭代 N（可重复多次）             │
├─────────────────────────────────────┤
│ [1] CodeGen                         │
│     输入: requirement + design + 失败反馈(可选)
│     输出: 代码补丁
│                                     │
│ [2] WorkspaceManager                │
│     创建隔离工作区                  │
│                                     │
│ [3] Patcher                         │
│     应用代码补丁                    │
│                                     │
│ [4] SDETAgent                       │
│     生成测试套件                    │
│                                     │
│ [5] Runner                          │
│     在 Docker/本地 执行测试         │
│                                     │
│ ┌─ 成功? ──→ [✅] 完成               │
│ └─ 失败? ──→ [6] TriageAgent        │
│                    ↓                 │
│            ┌──────────────┐         │
│            │ 失败诊断     │         │
│            └──────────────┘         │
│                    ↓                 │
│            ┌──────────────┐         │
│            │ RetryManager │         │
│            │ 继续重试?    │         │
│            └──────────────┘         │
│                ↓                     │
│            是 ──→ 注入反馈 ──→ 迭代 N+1
│            否 ──→ [❌] 停止
└─────────────────────────────────────┘
```

## 🧪 端到端测试场景

```python
{
    "requirement_raw": "这是一个计算器应用的增强需求：确保现有的加法函数正常工作并通过所有测试",
    "codebase": {
        "repo_path": "<testcode代码库>"
    }
}
```

### 测试输出（SelfHealingReport）

```python
{
    "success": bool,                    # 最终是否成功
    "iterations": int,                  # 迭代次数
    "final_code": Patch[],             # 最终代码补丁
    "test_results": {                   # 测试结果
        "passed": bool,
        "exit_code": int,
        "logs": str
    },
    "failure_history": FailureAnalysis[],  # 失败历史
    "final_failure": FailureAnalysis,   # 最终失败（如有）
    "total_time": float,                # 总耗时（秒）
    "iterations_log": SelfHealingIteration[]  # 迭代日志
}
```

## 🚀 使用示例

```python
from backend.agent.self_healing import SelfHealingCoordinator

# 初始化协调器
coordinator = SelfHealingCoordinator(
    max_retries=3,      # 最多重试 3 次
    use_docker=False,   # 使用本地执行或 Docker（可选）
)

# 执行自愈循环
report = coordinator.execute_with_self_healing({
    "requirement_raw": "完成一个 REST API 项目",
    "codebase": {
        "repo_path": "/path/to/project",
    }
})

# 查看结果
if report.success:
    print(f"✅ 成功！耗时 {report.total_time:.1f}s，迭代 {report.iterations} 次")
    print(f"应用的补丁: {len(report.final_code)} 个")
else:
    print(f"❌ 未成功。最后错误: {report.final_failure.error_type}")
    print(f"迭代次数: {report.iterations}/{3}")
    print(f"失败历史: {len(report.failure_history)} 条")

    # 查看每轮失败信息
    for i, failure in enumerate(report.failure_history, 1):
        error_type = failure.error_type if isinstance(failure.error_type, str) else failure.error_type.value
        print(f"  {i}. {error_type} (置信度: {failure.confidence:.0%})")
        print(f"     建议: {failure.suggestion}")
```

## 🎓 架构亮点

1. **单一职责** - 每个组件职责清晰
   - TriageAgent: 只负责诊断
   - RetryManager: 只负责决策
   - SelfHealingCoordinator: 只负责协调

2. **防循环设计** - 多重检查
   - 最大重试次数限制
   - 循环错误检测
   - 不可恢复错误过滤

3. **完整反馈链** - 信息流通畅
   - 失败 → 诊断 → 反馈 → 修复 → 重新测试

4. **清晰的数据模型** - 结构化输入输出
   - ErrorType 枚举：14 种错误分类
   - FailureAnalysis：完整的失败分析
   - SelfHealingReport：最终报告

5. **易于扩展** - 新增需求低成本
   - 添加新的错误类型只需修改 ErrorType
   - 添加新的诊断规则只需扩展 TriageAgent
   - 添加新的修复策略只需扩展 CodeGen feedback

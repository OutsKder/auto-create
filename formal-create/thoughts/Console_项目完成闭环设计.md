# Console 项目完成闭环设计

## 核心判断

用户不是来管理 Pipeline 的，用户是来完成一个项目。

所以 Console 的第一目标不是展示状态机，而是让用户舒服、清晰、可控地完成一次从需求到交付的闭环。

## 用户主流程

```text
输入项目目标
→ AI 生成交付计划
→ 用户确认计划
→ AI 执行各阶段
→ 用户审批关键节点
→ 生成最终交付包 / MR
```

这条流程要比页面结构更重要。所有角色视图、昼夜模式、审批卡片、结果页，都应该服务于这条闭环。

## Console 的三层结构

### 1. 项目目标层

顶部永远告诉用户：

- 当前在做哪个项目
- 当前处于什么状态
- 是否需要用户介入
- 当前角色视角是什么

### 2. 流水线层

中间展示 AI 执行路径，但不要只显示状态名。

应该让用户知道：

- AI 正在做什么
- 已经产出了什么
- 下一步会发生什么
- 卡在哪个审批点

后端状态机仍然保持：

```text
CREATED -> RUNNING -> WAITING_APPROVAL -> FINISHED
```

### 3. 当前动作层

右侧或下方展示用户此刻最该做的动作：

- 补充需求
- 确认交付计划
- 审批架构方案
- 退回并补充约束
- 查看交付包
- 进入 MR

用户每一刻都应该知道下一步，而不是自己找按钮。

## Role Lens 的真实作用

Role Lens 不是四个 Console，也不是四套流程。

它是一套 Console 上的四种读法：

- 第一次使用：解释状态、提示下一步、降低理解成本
- 产品 / 业务：突出需求理解、用户故事、边界条件、验收标准
- 技术负责人：突出架构方案、API、状态机、代码 Diff、测试报告、MR
- 管理者：突出进度、风险、阻塞点、审批状态、交付摘要

接口不变，状态机不变，数据不变。

变化的是：

- 文案
- 默认展开内容
- 信息排序
- 审批摘要重点
- 洞察卡指标

## Checkpoint 的设计原则

`WAITING_APPROVAL` 不能只是一个状态标签。

它应该主动告诉用户：

- 为什么这里停下来了
- AI 产出了什么
- 用户需要判断什么
- 批准后会发生什么
- 拒绝后会回到哪里

同一个 checkpoint，在不同角色下摘要不同：

- 新手：这里需要你确认方向，确认后 AI 会继续生成代码
- 产品：请检查用户故事和验收标准是否符合业务目标
- 技术：请检查架构边界、接口设计和测试覆盖
- 管理者：当前审批阻塞交付，处理后预计进入 MR 阶段

## 按钮文案原则

不要使用泛化按钮：

- 提交
- 运行
- 查看
- 下一步

应该使用动作明确的按钮：

- 生成交付计划
- 确认计划并启动
- 批准架构方案
- 退回并补充约束
- 查看交付包
- 打开 MR

按钮应该描述点击后系统会发生什么。

## Console 结构建议

```text
顶部栏
- 项目名
- 当前状态
- Role Lens
- 昼夜切换

左侧
- 项目 / 流水线列表
- 制品库
- 审批中心

主区域
- 项目启动器 / 当前流水线
- 状态轨道
- 当前节点详情

右侧
- 当前建议动作
- 角色视角摘要
- 风险 / 审批 / 产物入口
```

## 实现优先级

1. 先完成 Role Lens，让同一 Console 能按角色改变信息重点。
2. 再设计项目完成闭环，让用户从输入目标走到交付包。
3. 再做 Console 昼夜模式，让日间适合审阅，夜间适合 AI 运行态。
4. 最后补 Checkpoint / Result 的高保真交互。

## 一句话原则

Console 不是后台面板，而是一台项目交付驾驶舱。

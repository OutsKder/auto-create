# 织界引擎 MVP：后端（FastAPI）技术实现与接口方案

作为织界引擎的项目主管，为了确保我们在最短时间内交付一个**稳定、可演示、符合 AI Native 特性且极具技术说服力**的 MVP 参赛版本，特制定本基于 Python + FastAPI 的后端技术实现方案。

本方案的核心宗旨是：**AI 原生开发、流式响应、数据结构化约束与防翻车熔断**。

---

## 1. 技术栈选型（Tech Stack）

考虑到我们是在做一个带有强烈 AI 属性的“多智能体工作流引擎”，选用 Python 后端生态将能让我们零阻力地接入各类 AI 基础设施。

| 模块 | 技术选型 | 选用原因（Why） |
| :--- | :--- | :--- |
| **框架与运行时** | **FastAPI + Uvicorn** | 全异步并发，极速轻量；自带自动生成的 Swagger API 文档，比赛展示极具专业感。 |
| **数据库层** | **SQLite + SQLModel / SQLAlchemy** | **零配置单文件存储**，不依赖外部环境，并且通过 ORM 免写 SQL 即可完成对象存取。 |
| **大模型通信** | **SSE (Server-Sent Events) via StreamingResponse** | FastAPI 原生优雅支持流式响应（`yield`），完美实现打字机特效；前端原生 `EventSource` 即可接收。 |
| **AI 约束与骨架** | **Pydantic** | 强制大模型输出结构化数据（如 JSON），并在接口层直接完成格式校验，拦截 AI 幻觉，保证“流程引擎”的确定性。 |
| **AI 接入层** | **openai-python / LangChain** | 调大模型的业界绝对标准库，方便后续无缝替换普通或支持思考过程的各种模型。 |

---

## 2. 快速与稳定交付策略（How to be Fast & Stable）

比赛 Demo 最大的痛点是现场网络卡顿或 API 限流导致“转圈圈”。我们必须引入**兜底与防翻车机制**。

1. **Mock-First 伪数据打底**：
   - 第一周：在 FastAPI 的 Controller 中，直接 `yield` 写死的一套极其专业的 Markdown 节点数据，先把前端 API 和状态机流转跑通。
   - 第二周：逐渐把这些伪接口换成向大模型发起真实推流的接口。
2. **熔断与降级机制 (Fallback)**：
   - 所有接口支持 `?mock=true` 参数。准备展示时，可动态切换到 Mock 模式，自动熔断真实的外部请求，瞬间返回精美的假数据，确保演示 100% 安全。
3. **前端低侵入改造**：
   - 前端原生逻辑不改，剥离 `localStorage`，统一封装一个类似 `apiClient` 的工具类发送 `fetch` 即可。

## 3. 前后端接口实现详案 (API & Integration)

使用 FastAPI 时，我们的路由非常语义化。这里重点说明前后端的协议。

### 3.1 实体模型 (Pydantic / DB Schema)
- `Requirement`: 需求模型，包含基础字段、流程状态（`status: PENDING | RUNNING | WAITING_REVIEW | COMPLETED`）。
- `StageLog`: 各 Agent 阶段的响应日志和审核状态。

### 3.2 REST API 契约协议

#### A. 创建需求 & 初始化 Pipeline
- **URL**: `POST /api/v1/pipeline/start`
- **Request Body**:
  ```json
  { "title": "新增审批中心", "background": "...", "priority": "high" }
  ```
- **后端处理**: 将内容存入 SQLite，生成并返回 `requirement_id`，初始化各个执行节点。
- **前端响应**: 调用此请求后，UI 直接渲染该流程进入“仪表盘”。

#### B. 人工审核节点（人类大脑介入）
- **URL**: `POST /api/v1/pipeline/{id}/stage/{stage_id}/audit`
- **Request Body**:
  ```json
  { "action": "approve", "note": "UI需参考苹果规范" } // 或 "reject"
  ```
- **后端处理**: 更新审核状态。若通过，则开放下一节点的调用权限。

#### C. 🔥 触发 Agent 执行与流式输出（核心引擎接口）
- **URL**: `GET /api/v1/pipeline/{id}/stage/{stage_id}/execute/stream`
- **协议**: Server-Sent Events (SSE), FastAPI 的 `StreamingResponse`。
- **前后端交互流程**:
  1. Frontend 使用原生 API 发起：
     ```javascript
     const eventSource = new EventSource(`/api/v1/pipeline/${reqId}/stage/solution/execute/stream`);
     ```
  2. Backend 在 FastAPI 中使用 `yield` 返回流：
     ```python
     async def fake_llm_stream():
         for word in "AI 正在生成技术方案...":
             yield f"data: {json.dumps({'status': 'running', 'text': word})}\n\n"
             await asyncio.sleep(0.1)  # 模拟延迟
         # 结束标记
         yield f"data: {json.dumps({'status': 'waiting_review', 'text': ''})}\n\n"
     
     return StreamingResponse(fake_llm_stream(), media_type="text/event-stream")
     ```
  3. Frontend 监听 `onmessage`：每次触发时，将获取到的 `text` 增量附加到界面的 Markdown 可视化框里，实现沉浸式“打字机”特效。
  4. 当前端收到 `status: 'waiting_review'` 时，关闭 `eventSource`，页面点亮黄灯，展现人脑拦截（通过/驳回）按钮。

## 4. 行动路线部署 (Action Items)

1. **[Backend] 基建搭建 (1 天)**：建立 FastAPI 项目结构 (`main.py` + `models.py` + `routes.py`)，写通硬编码的 Mock 数据接口。
2. **[Frontend] 路由解耦 (2 天)**：在 `app.js` 里编写统一的 API 调用层，使用 `fetch` 和 `EventSource` 替代原有的 `localStorage` 操作。
3. **[Fullstack] SSE 打字机流转攻坚 (1 天)**：实现前端页面上字符一行一行出现的流水线自动推进体验。
4. **[Backend] AI 赋能**：在 FastAPI 后端中接入真正的 OpenAI 兼容库，传入提示词让 AI 真的接盘工作。
